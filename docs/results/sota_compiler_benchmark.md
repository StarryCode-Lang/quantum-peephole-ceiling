## 3. Pipeline corrections applied on 2026-07-20 (provenance)

Two latent bugs in `experiments/sota_benchmark.py` were found and fixed during the audit
(version bump 1.0.0 → 1.1.0):

1. **Cirq CZTargetGateset step never ran.** `optimize_for_target_gateset(...,
   target_gateset=...)` raised `TypeError` on cirq 1.6.1 (keyword renamed to `gateset=`)
   and the bare `except` silently skipped the step. The 2026-07-18 Cirq data therefore
   reflects an incomplete pipeline. Fixed and rerun: `cirq_default_20260720_151000.csv`
   (520/520 ok). The protocol-conformant pipeline changes Cirq's numbers substantially
   (e.g. CNOT 0.0 → 100.0%, RandomClifford 47.4 → 63.1%, QuantumWalk −382.6 → −4547.4%)
   because the CZ-target rewriting now actually executes.
2. **Cirq QASM round-trip failures.** Cirq's QASM export emits `sx`/`sxdg`, which
   `qiskit.qasm2`'s builtin qelib1 set does not define — 100 rows (Grover/QFT/QuantumWalk)
   failed to re-import on 2026-07-18. Fixed by injecting the standard qelib1 definitions
   (`gate sx a { sdg a; h a; sdg a; }`, `gate sxdg a { s a; h a; s a; }`) after the include
   line. Zero import errors in the rerun.

Aggregation fixes: canonical-file selection now keys on (tool, config) parsed as
`{tool}_{config}_{date}_{time}` (config may contain underscores) — previously the date
leaked into the key, mixing smoke/buggy/fixed Cirq runs into one aggregate; fidelity pass
rate is now computed over exact-fidelity rows only; success rate is computed over all
rows including errors/timeouts; aggregated output is written atomically with a
timestamped backup of the previous file.

## 4. Leading finding: production compilers exceed the prototype on 7 of 15 ceiling families

The manuscript's Class-III statement is confirmed by the canonical data with two
quantified caveats. On the 10 families where the peephole prototype sits at its
action-space ceiling (~0% reduction), production compilers split them into:

- **3 genuine structural ceilings** (QFT, GHZ, SurfaceCode): every tool achieves ~0%
  (Qiskit 0.0/0.0/0.0; t|ket> −206.7/0.0/0.0; Cirq −258.0/−181.9/+8.4).
- **7 prototype-limited ceilings** (QAOA, VQE, HardwareEfficient, IQP, UCCSD, Adder,
  QuantumWalk): the prototype's 0% is *its own* limit, not a structural one. For **5 of
  the 7**, a production compiler exceeds the prototype by 6–64 pp (best tool in
  parentheses): IQP +64.4 (t|ket>), VQE +42.2 (t|ket>), HardwareEfficient +36.5 (t|ket>),
  UCCSD +23.3 (Qiskit), QAOA +6.3 (t|ket>). For the **remaining 2**, every tested
  production compiler also fails: Adder is 0.0 (Qiskit), −505.9 (t|ket>), −1217.3 (Cirq);
  QuantumWalk is negative for *all* tools (Sec. 6). The audit therefore recommends
  flagging Adder and QuantumWalk as *practical* ceilings under current compilers —
  QuantumWalk being the clearest case of a production compiler making circuits strictly
  worse — rather than as families where production wins.

Beyond the ceiling families, production compilers also dominate where the prototype is
weakly positive: CNOT 66.7 → 100.0, Oracle 23.1 → 41.8, RandomClifford 12.3 → 72.5,
HaarRandom 0.0 → 6.5 (all pp gaps vs the best production tool, E15 prototype at
n in {4,6,8} vs v6 production). Counting every family where the best production compiler
beats the prototype by ≥ 5 pp gives **8 of 15** (CNOT, HaarRandom, HardwareEfficient,
IQP, Oracle, QAOA, RandomClifford, VQE; UCCSD makes 9 at +23.3 pp with the naming
unification). The "7/15" headline specifically refers to the ceiling-class families.

Complementary finding: **the prototype never increases gate count** (minimum observed
reduction 0.0% across all datasets), while every production compiler blows up on at
least 2 families (Qiskit 2, t|ket> 4, Cirq 10 of 15). This asymmetry is the practical
argument for ceiling-aware compilation: skip or gate the optimization when the
structural ceiling is zero or when decomposition dominates.

## 5. Qiskit L0–L3 progression (new 2026-07-20 data)

Fair-mode level sweep (520 rows per level, 10 trials/cell, deterministic across repeat
executions):

- **L2 ≡ L3**: identical optimized gate counts on 100% of 520 matched rows. In fair mode
  the L3-only passes (Collect2qBlocks/ConsolidateBlocks) add nothing over L2 on this
  suite; L3 can be replaced by L2 at equal quality and lower pass overhead.
- **L1 < L2** on 30.8% of rows, concentrated in IQP (+29.8 vs +62.8), RandomClifford
  (+61.6 vs +72.5), UCCSD (+11.0 vs +23.3), HaarRandom, Grover, QuantumWalk — consistent
  with CommutativeCancellation being scheduled only from L2.
- **L0** performs no optimization (0% on 13 families) yet still inflates Grover (−675.7%)
  and QuantumWalk (−2287.0%): HighLevelSynthesis decomposes multi-controlled gates even
  at level 0. "Optimization level 0" is therefore not a gate-count-neutral baseline for
  circuits containing multi-controlled operations.

Cross-validation with legacy E12 (1 trial/cell): identical level ordering and the same
L2 ≡ L3 pattern; per-family means agree within sampling noise.

## 6. QuantumWalk −256.5%: mechanism of the negative reduction

At n = 4 the canonical Qiskit L3 run maps 46 gates → 164 gates, i.e. **−256.5%**
(the same value appears in the pass-isolation Table 2). The blowup escalates with size:
−681.0% (n = 5), −302.9% (n = 6), −2704.9% (n = 7), −3289.4% (n = 8), −4193.4% (n = 9).

Decomposition of the n = 4 blowup: CNOT count 6 → 75 (12.5×), total depth 29 → 149.
Mechanism:

1. The coined-walk increment operators are **multi-controlled gates**. Qiskit's
   transpiler — at *every* level including L0 — first runs HighLevelSynthesis, which
   expands each multi-controlled gate into the 1q/2q basis. t|ket>'s DecomposeBoxes and
   Cirq's CZTargetGateset do the equivalent.
2. The decomposed form has **no adjacent inverse pairs and no exploitable commutation
   structure**, so the peephole passes that follow find nothing to cancel; the
   one-time decomposition cost is pure gate-count inflation.
3. The transformation is **exact**: F_avg = 1.0 on every QuantumWalk row for every tool.
   The failure is metric-specific (gate count), not correctness.

QuantumWalk is thus the canonical counterexample to "production compiler ⇒ gate-count
safe": on decomposition-heavy structured circuits, all three production pipelines make
the circuit strictly larger — Qiskit −1904.7%, Cirq −4547.4%, t|ket> −1261.9% pooled —
while the ceiling-aware prototype correctly leaves the circuit unchanged (0.0%).

## 8. Reliability: fidelity and success rates

- **Qiskit** (all levels): 100% success (520/520), F_avg ≥ 0.999 on 100% of the 480
  exact-fidelity rows (n ≤ 8).
- **Cirq** (fixed pipeline): 100% success (520/520, zero of the former 100 import
  errors), F_avg ≥ 0.999 on 100% of 396 exact rows; min F = 0.9999999999995.
- **t|ket>**: 100% compiler success (440/440), but **14 of 30 RandomClifford trials fail
  exact fidelity** with F_avg ≈ 0.27–0.33 — precisely |Tr(U1†U2)| = d/2 for d = 8/16/32,
  i.e. the optimized unitary differs from the target on half the computational basis.
  Failures are bimodal (F = 1.0 or F ≈ 0.3), seed-dependent, and confined to
  RandomClifford; all other 14 families pass at F = 1.0. The nominal t|ket>
  RandomClifford reduction (+71.6%) is therefore partially invalid; on fidelity-passing
  rows it is +73.5% (n = 16). This is a reproducible pytket 2.18.0 correctness issue on
  random Clifford circuits and should be disclosed wherever the t|ket> RandomClifford
  number is cited.

Read together with the full-pipeline results (Sec. 4–5):

1. **The peephole horizon is real and narrow.** One isolated Qiskit pass
   (CommutativeCancellation) reproduces the prototype's entire reduction on CNOT,
   Oracle, and nearly all of it on RandomClifford (25.3% vs 23.8%). The prototype's

## 12. t|ket> RandomClifford correctness incident: root cause and disposition (wave 4)

**Status update (2026-07-21).** The wave-1 audit (Sec. 8) recorded 14/30 t|ket>
RandomClifford trials failing exact fidelity. Root cause has now been identified and
reproduced in isolation; the disposition below supersedes the open handoff item 3 of
`docs/review/wave1/compiler_baseline.md`.

### 12.1 Phenomenon (canonical data)

In `data/v6/sota_benchmark/raw/tket_default_20260718_173443.csv`, 14 of the 30
RandomClifford rows (n = 3–5, 10 trials each) have F_avg in {1/3, 5/17, 3/11} =
(d/4 + 1)/(d + 1) for d = 2^n — i.e. **|Tr(U_in† U_out)| = d/2 exactly** — while the
remaining 16 rows pass at F_avg = 1.0. The failures are bimodal, seed-dependent, and
confined to RandomClifford; all other 14 families pass at F = 1.0. The nominal
t|ket> RandomClifford reduction (+71.6%) is partially invalid; on fidelity-passing
rows it is +73.5% (n = 16).

### 12.2 Root cause (identified 2026-07-21)

Not an arithmetic error inside FullPeepholeOptimise's resynthesis, but an
**implicit-permutation semantics hazard at the pytket→qiskit boundary**:

1. `FullPeepholeOptimise` defaults to `allow_swaps=True` (installed docstring:
   "whether to allow implicit wire swaps"). On the failing trials it eliminates
   2-qubit gates by permuting the wire assignment, leaving an **implicit qubit
   permutation** on the circuit (e.g. n = 3 trial 2: q[1]↔q[2]; n = 4 trial 4:
   q[0]↔q[1]; n = 5 trial 4: q[2]↔q[3] — always a single transposition).
2. `tk_to_qiskit` (pytket-qiskit 0.77.0) does **not** materialize the permutation;
   it emits only a `UserWarning` ("The pytket Circuit contains implicit qubit
   permutations which aren't handled by default…") and returns a circuit that
   implements U_in·P instead of U_in. A single transposition P has exactly d/2 fixed
   points, hence |Tr(U_in† U_out)| = |Tr(P)| = d/2 — the observed fingerprint.
3. Verification: calling `Circuit.replace_implicit_wire_swaps()` before
   `tk_to_qiskit` restores **F_avg = 1.0 on all three examined failing cells**;
   running `FullPeepholeOptimise(allow_swaps=False)` also yields F_avg = 1.0 on all
   five examined failing cells, at a cost of 2–5 additional gates per circuit
   (e.g. n = 5 trial 4: 10 → 15 gates). The optimized gate counts in the canonical
   CSV are therefore real tket outputs, mildly optimistic because the elided final
   permutation is not charged as SWAP gates.

### 12.4 Disposition decision

**Keep the canonical data as-is and disclose prominently** (footnote wherever the
t|ket> RandomClifford number is cited; the manuscript fragment already carries the
auto-computed caveat). Rationale:

1. The incident is a faithful record of how a production toolchain behaves under a
   naive-but-standard integration path (defaults on both sides); documenting the
   trap has standalone reporting value for practitioners.
2. The gate-count measurements are not fabricated or buggy on our side — they are
   tket's real outputs; only the fidelity *comparison* was confounded by the dropped
   permutation, and the confound is now fully explained and quantified
   (|Tr| = d/2 ⇔ single elided transposition).
3. Rewriting history (silent rerun with `allow_swaps=False`) would destroy the
   provenance trail; instead, future t|ket> runs should apply
   `replace_implicit_wire_swaps()` before `tk_to_qiskit`, or use
   `allow_swaps=False`, and the 2–5-gate swap-elision discount should be reported
   alongside.
4. Numerical consequence is bounded: fidelity-passing mean +73.5% vs nominal +71.6%;
   with `allow_swaps=False` the reduction is a few points lower still on failing
   cells. None of the Sec. 4–6 findings (structural ceilings, QuantumWalk blowup,
   L0–L3 progression) involves RandomClifford t|ket> fidelity.

### 12.5 Draft bug report for Quantinuum (pytket-qiskit)

> **Title**: `tk_to_qiskit` silently drops implicit qubit permutations: returned
> circuit is unitarily inequivalent (warning-only)
>
> **Environment**: pytket 2.18.0, pytket-qiskit 0.77.0, qiskit 2.4.1, Python 3.12,
> Windows 11 (also expected on other platforms).
>
> **Summary**: When a pytket circuit carries an implicit qubit permutation (produced
> e.g. by `FullPeepholeOptimise` with its default `allow_swaps=True`),
> `pytket.extensions.qiskit.tk_to_qiskit` returns a qiskit circuit that implements
> U·P (P = the implicit permutation) rather than U. The only signal is a
> `UserWarning`, which is invisible in non-interactive pipelines. For a single
> transposition the mismatch is large: |Tr(U†U_out)| = d/2, i.e. average gate
> fidelity ≈ (d/4+1)/(d+1) ≈ 0.27–0.33 for d = 8–32.
>
> **Minimal reproducer** ( Clifford circuit, n = 5 ):
> ```python
> import numpy as np
> from qiskit import QuantumCircuit
> from qiskit.quantum_info import Operator
> from pytket.extensions.qiskit import qiskit_to_tk, tk_to_qiskit
> from pytket.passes import FullPeepholeOptimise
>
> rng = np.random.RandomState(4847)
> qc = QuantumCircuit(5)
> for _ in range(10):
>     for q in range(5):
>         g = rng.choice(["h", "s", "id"])
>         if g == "h": qc.h(q)
>         elif g == "s": qc.s(q)
>     if rng.random() < 0.5:
>         c = rng.randint(0, 4); qc.cx(c, c + 1)
>
> tk = qiskit_to_tk(qc)
> FullPeepholeOptimise().apply(tk)          # default allow_swaps=True
> print(tk.implicit_qubit_permutation())    # {q[2]: q[3], q[3]: q[2], ...}
> out = tk_to_qiskit(tk)                    # only a UserWarning
> U, Uo = Operator(qc).data, Operator(out).data
> tr = abs(np.trace(U.conj().T @ Uo))       # 16.0 == d/2  (expected 32.0)
> tk.replace_implicit_wire_swaps()
> Uf = Operator(tk_to_qiskit(tk)).data
> assert np.allclose(U, Uf, atol=1e-8)      # passes after explicit materialization
> ```
>
> **Expected behavior**: `tk_to_qiskit` should default to
> `replace_implicit_swaps=True` (or raise when a nontrivial implicit permutation is
> present). A warning-only path that returns a unitarily inequivalent circuit is a
> correctness trap: every downstream consumer that trusts the converted circuit
> (fidelity checks, simulators, hardware submission) silently computes the wrong
> unitary.
>
> **Actual behavior**: warning only; returned circuit differs from the input by a
> trailing qubit permutation; measured average gate fidelity as low as 0.27 on
> n = 3–5 random Clifford circuits (14/30 seed-dependent failures in our 30-trial
> sweep).
>
> **Notes**: pytket documents implicit swaps and offers both
> `replace_implicit_wire_swaps()` and the `replace_implicit_swaps` conversion flag,
> so this is a default-safety issue rather than missing functionality. We hit it via
> the entirely default path `qiskit_to_tk` → `FullPeepholeOptimise()` →
> `tk_to_qiskit`.
